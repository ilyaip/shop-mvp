# Документ требований: Система динамической темизации и улучшения UI/UX

## Введение

Система динамической темизации превращает интернет-магазин в конструктор, где все визуальные настройки (цвета, логотип, название бренда, изображения) загружаются с бэкенда. Это позволяет администратору полностью кастомизировать внешний вид магазина без изменения кода. Также включает улучшения UI/UX с элементами дизайна в стиле Apple (liquid glass эффекты) и новую посадочную страницу.

## Глоссарий

- **Theme**: Набор визуальных настроек (цвета, шрифты, изображения)
- **FrontSettings**: Объект с настройками темы, получаемый с бэкенда
- **ThemeStore**: Pinia хранилище для управления темой
- **Landing_Page**: Посадочная страница с приветствием и описанием магазина
- **Liquid_Glass**: Стиль дизайна с эффектом матового стекла (backdrop-filter, blur)
- **Theme_Switcher**: UI компонент для переключения между предустановленными темами (для демо)
- **Brand_Settings**: Настройки бренда (логотип, название, описание)

## Требования

### Требование 1: Система динамической темизации

**User Story:** Как администратор, я хочу настраивать внешний вид магазина через бэкенд, чтобы создавать уникальный брендинг без изменения кода.

#### Критерии приемки

1. THE System SHALL создать хранилище `useThemeStore` для управления настройками темы
2. WHEN приложение загружается, THE System SHALL загружать настройки темы из API (моковые данные)
3. THE System SHALL применять цветовые настройки через CSS-переменные динамически
4. THE System SHALL поддерживать следующие цветовые настройки: primary, secondary, background, text, border, success, error
5. WHEN настройки темы изменяются, THE System SHALL обновлять CSS-переменные в реальном времени
6. THE System SHALL сохранять загруженные настройки в памяти приложения

### Требование 2: Настройки бренда

**User Story:** Как администратор, я хочу настраивать логотип, название и описание магазина, чтобы создать уникальную идентичность бренда.

#### Критерии приемки

1. THE System SHALL загружать настройки бренда из API: логотип (URL), название, описание
2. WHEN настройки бренда загружены, THE System SHALL отображать логотип в шапке сайта
3. WHEN настройки бренда загружены, THE System SHALL отображать название бренда в шапке и title страницы
4. THE System SHALL использовать описание бренда на посадочной странице
5. THE System SHALL предоставлять fallback значения если настройки не загружены

### Требование 3: Посадочная страница (Landing Page)

**User Story:** Как пользователь, я хочу видеть приветственную страницу с информацией о магазине, чтобы понять концепцию бренда перед покупками.

#### Критерии приемки

1. THE System SHALL создать новую посадочную страницу по маршруту `/`
2. THE System SHALL отображать hero-секцию с фоновым изображением из настроек
3. THE System SHALL отображать название и описание бренда из настроек
4. THE System SHALL отображать кнопку "Перейти в каталог" с переходом на `/catalog`
5. THE System SHALL применять liquid glass эффекты к элементам посадочной страницы
6. THE System SHALL обеспечить адаптивность посадочной страницы для всех устройств
7. WHEN пользователь переходит на `/catalog`, THE System SHALL отображать каталог товаров

### Требование 4: Улучшения UI/UX - Liquid Glass дизайн

**User Story:** Как пользователь, я хочу видеть современный интерфейс с эффектами матового стекла, чтобы получить премиальный опыт использования.

#### Критерии приемки

1. THE System SHALL применять backdrop-filter с blur эффектом к шапке сайта
2. THE System SHALL применять полупрозрачный фон к шапке с эффектом матового стекла
3. THE System SHALL применять liquid glass эффекты к карточкам товаров при hover
4. THE System SHALL применять liquid glass эффекты к модальным окнам
5. THE System SHALL использовать плавные анимации для всех интерактивных элементов
6. THE System SHALL обеспечить производительность эффектов на всех устройствах

### Требование 5: Улучшения карточек товаров

**User Story:** Как пользователь, я хочу видеть привлекательные карточки товаров с закругленными углами, чтобы интерфейс выглядел современно.

#### Критерии приемки

1. THE System SHALL увеличить border-radius превью изображений товаров до 16px
2. THE System SHALL применять тени к карточкам товаров для создания глубины
3. THE System SHALL добавить плавную анимацию scale при hover на карточку
4. THE System SHALL добавить плавную анимацию для изображений товаров
5. THE System SHALL обеспечить консистентность стиля всех карточек

### Требование 6: Удаление фильтров по категориям

**User Story:** Как пользователь, я хочу видеть упрощенные фильтры без категорий, так как все товары из одной категории.

#### Критерии приемки

1. THE System SHALL удалить компонент CategoryFilter из интерфейса
2. THE System SHALL удалить фильтрацию по категориям из useProductsStore
3. THE System SHALL сохранить фильтрацию по диапазону цен
4. THE System SHALL обновить дизайн секции фильтров для отображения только фильтра цен
5. THE System SHALL обеспечить адаптивность обновленных фильтров

### Требование 7: Переключатель тем (для демо)

**User Story:** Как разработчик, я хочу иметь возможность переключения между предустановленными темами для демонстрации и отладки.

#### Критерии приемки

1. THE System SHALL создать компонент ThemeSwitcher на странице профиля
2. THE System SHALL предоставить минимум 3 предустановленные темы: Light, Dark, Blue
3. WHEN пользователь выбирает тему, THE System SHALL применять соответствующие цветовые настройки
4. THE System SHALL отображать текущую активную тему в ThemeSwitcher
5. THE System SHALL применять изменения темы мгновенно без перезагрузки страницы

### Требование 8: Моковые данные для настроек

**User Story:** Как разработчик, я хочу использовать моковые данные для настроек темы, чтобы разработать и протестировать функциональность до интеграции с бэкендом.

#### Критерии приемки

1. THE System SHALL создать моковые данные FrontSettings с полным набором настроек
2. THE System SHALL включать в моковые данные: цвета, логотип, название бренда, описание, hero-изображение
3. THE System SHALL создать минимум 3 варианта моковых настроек для разных тем
4. THE System SHALL предоставить API функцию для загрузки моковых настроек
5. THE System SHALL имитировать асинхронную загрузку с задержкой

### Требование 9: Обновление маршрутизации

**User Story:** Как пользователь, я хочу чтобы главная страница была посадочной, а каталог был доступен по отдельному маршруту.

#### Критерии приемки

1. THE System SHALL изменить маршрут `/` на отображение посадочной страницы
2. THE System SHALL сохранить маршрут `/catalog` для каталога товаров
3. THE System SHALL обновить навигационные ссылки в шапке
4. THE System SHALL обновить все внутренние ссылки на каталог
5. THE System SHALL обеспечить корректную работу навигации между страницами

### Требование 10: Дополнительные улучшения UI/UX

**User Story:** Как пользователь, я хочу видеть улучшенный интерфейс с современными элементами дизайна.

#### Критерии приемки

1. THE System SHALL улучшить spacing между элементами для лучшей читаемости
2. THE System SHALL добавить плавные transitions для всех интерактивных элементов
3. THE System SHALL улучшить типографику с использованием font-weight variations
4. THE System SHALL добавить subtle анимации при загрузке страниц
5. THE System SHALL улучшить визуальную иерархию элементов
6. THE System SHALL обеспечить консистентность дизайна на всех страницах

### Требование 11: Производительность и оптимизация

**User Story:** Как пользователь, я хочу чтобы приложение работало быстро даже с визуальными эффектами.

#### Критерии приемки

1. THE System SHALL оптимизировать backdrop-filter для производительности
2. THE System SHALL использовать CSS transforms для анимаций (GPU acceleration)
3. THE System SHALL ленивую загрузку для hero-изображения на посадочной странице
4. THE System SHALL минимизировать reflows при применении темы
5. THE System SHALL обеспечить плавность анимаций 60 FPS

### Требование 12: Обратная совместимость

**User Story:** Как разработчик, я хочу сохранить работоспособность существующего функционала при добавлении новых фич.

#### Критерии приемки

1. THE System SHALL сохранить работу всех существующих страниц (product, cart, checkout, profile)
2. THE System SHALL сохранить работу всех существующих stores (cart, products, user, ui)
3. THE System SHALL сохранить работу всех существующих компонентов
4. THE System SHALL обеспечить fallback на дефолтные значения если настройки не загружены
5. THE System SHALL не нарушать существующие тесты
